# DW 开发规范

## 指令优先级

1. 当前明确指令 > 仓库级规则
2. 仓库 `AGENTS.md` / `CLAUDE.md` 次之
3. 无冲突时，默认套用本 DW 规范

## 编码规则

- 源码默认使用 **ASCII**，避免直接书写原始中文
- 中文使用 Unicode 转义：

| 文件类型 | 转义格式 | 示例 |
|---------|---------|------|
| JS / TS / JSON / YAML | `\uXXXX` | `\u4e2d\u6587` |
| HTML / XML | `&#xXXXX;` | `&#x4E2D;&#x6587;` |

- **同一文件内不可混用**原始中文和转义中文，必须统一风格
- 新增文件一律用转义格式；存量文件暂不强制改造，但修改时逐步统一

## 代码结构规则

### 模块粒度

- 强调**小模块、单一职责**
- 拒绝"超级文件"硬塞功能
- **400+ 行**的文件在加新职责前，优先考虑拆分
- **800+ 行**的文件原则上禁止继续堆功能

### Electron 分层要求

- `preload` — 公共接口稳定，只暴露必要的 API
- `IPC` — 按域拆分注册，不要全堆在一个文件
- `service` — 业务逻辑放服务层
- `renderer` — 渲染层只负责 UI 展示
- `persistence` — 持久化逻辑独立
- `feature-center` — 相关逻辑严格按边界拆开

### 纯逻辑抽离

- 纯计算、纯规则、无副作用的逻辑，抽成**可单独验证的小模块**
- 便于单元测试和复用

## 数据组织规则

### 目录结构

- 按 **功能/模块/实体** 分目录，禁止全塞一起
- 配置（config）、缓存（cache）、状态（state）严格分开
- 持久化路径整体布局：

```
TEMU_Data_Electron/
├── <module>/config/
├── <module>/cache/
├── <module>/state/
└── TEMU_Data/
    └── <feature-scope>/
```

### 账号隔离

- 涉及账号的模块，本地缓存、云端数据、状态文件**必须按账号隔离**
- 不可跨账号混用同一个缓存或状态文件

## 用户文案规则

- UI 文案必须**面向用户**，用用户能理解的自然语言
- 严禁将内部实现细节暴露给用户，包括：
  - IPC 通道名
  - 异常堆栈 / 错误码
  - 内部模块名、函数名
- 文案要**短**，长解释走 `tooltip` / `popover`
- 提示信息要有实际指导意义，不是"操作失败"就完了

## 窗口和布局规则

- **简洁、平衡、专业**
- 能一行放下的内容，尽量一行
- 组件尺寸和视觉重量要统一，避免忽大忽小
- 弹窗默认**不可点遮罩关闭**（除非明确要求可关闭）

## 桌面 Keep-Alive 规则

- Electron 多模块切换默认采用**隐藏/显示**模式，而非销毁重建
- 切回来时保留原有状态（表单数据、滚动位置、选中项等）
- 长任务切到别的模块后**继续运行**，不中断
- 窗口最小化/隐藏后，后台任务默认不停止
- 禁止偷懒用 `location.reload()` 代替状态恢复

## Electron 调试规则

- 不要以为重新打开窗口就等于代码生效
- `main` / `preload` 改动后必须**真正重启进程**
- 调试前确认没有旧进程残留（检查 Task Manager / `ps`）
- 必要时加**短期运行日志**确认真实数据流，调试完毕删除
- 区分 Vite HMR（renderer 热更新）和 Electron 重启（main/preload）

## 浏览器自动化规则

- 以网络返回的 `taskId` / `submitId` / `result URL` 为**权威判断依据**
- **不要靠 DOM 变化猜测**操作是否成功
- 页面 helper 脚本尽量一次注入、重复复用
- 高频日志必须**节流**和**摘要化**，避免刷屏
- 能用 `session.fromPartition(...).fetch(...)` 就别用页面注入 `fetch`
- 店铺级别的请求必须绑定正确的 `partition`，不可 fallback 到 `defaultSession` 或全局 `fetch`

## 命令执行规则

- 构建、验证、排查**一步一步跑**
- 每一步看结果再决定下一步
- 不要把所有无关命令串成一大坨
- 中间某步出错，先排查修复再继续
